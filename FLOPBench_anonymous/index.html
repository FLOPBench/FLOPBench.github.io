gpuFLOPBench

A benchmark and analysis site for studying whether off-the-shelf LLMs can provide Roofline-oriented GPU triage by predicting profiler-derived arithmetic intensity before scarce accelerator time is available.

Read motivation Explore performance Open explorer

Why this project exists

  • Motivation. Profiler-driven GPU optimization breaks down when target hardware is unavailable, rationed, or too costly for early-stage triage.
  • Signal. Arithmetic intensity gives an interpretable Roofline quantity for deciding whether effort should focus on memory traffic or computation.
  • Goal. The site exposes the corpus used to test whether prompting alone is enough for modern LLMs to approximate that signal across multiple GPUs.
Research framing

Motivation, key question, and the questions that drive the study.

The project studies whether a smaller, interpretable performance signal can guide GPU optimization before full profiler feedback is available. It positions arithmetic intensity as an early triage quantity rather than a replacement for hardware profiling.

motivation

Profiler-driven optimization does not scale to every iteration.

Typical LLM-based GPU optimization loops depend on repeated execution and profiling of candidate variants. That coupling makes optimization quality dependent on accelerator availability, which is often the scarcest resource during early kernel triage.

key question

Can prompting alone recover a useful Roofline signal?

The central question is whether off-the-shelf state-of-the-art LLMs, without task-specific training or fine-tuning, can act as GPU Roofline triage tools by predicting a kernel's arithmetic intensity.

research questions

Four questions structure the evaluation.

  • RQ1. Can LLMs predict arithmetic intensity well across multiple GPU architectures?
  • RQ2. Does agentic prompting outperform zero-shot prompting for this task?
  • RQ3. Which source-code features lead to larger prediction errors?
  • RQ4. Do post-compilation SASS artifacts improve prediction quality?
contributions

What the study contributes.

  • Zero-training test. It evaluates whether prompting alone can estimate profiler-derived arithmetic intensity across multiple GPU targets.
  • Prompting tradeoff. It compares zero-shot and agentic prompting to determine when extra reasoning cost is worthwhile.
  • Artifact value. It measures how much SASS helps when source-only reasoning misses compiler-realized behavior.
  • Triage framing. It positions LLM-based static profiling as a complement to, not a substitute for, hardware profilers.
Benchmark structure

One benchmark, multiple layers of visibility.

gpuFLOPBench is not just a single score. It exposes the benchmark from the inventory level down to exact kernel rows so that category coverage, roofline behavior, and source-level detail stay visible at the same time.
